# AnswerRush — Project Documentation
### CatHacksXII Hackathon Submission

---

## Overview

AnswerRush is a real-time multiplayer quiz brawler built for the CatHacksXII 24-hour hackathon. Up to 4 players join the same game room and compete to answer trivia questions. Each round, a question appears and 4 answer platforms are placed across the level. Players race, jump, and fight each other Stick Fight-style to reach the platform they think is correct. Whoever is standing on the correct platform when the 15-second timer runs out earns a point. First player to accumulate the most points after 15 rounds wins.

The game runs entirely in the browser — no installation required. Players join by opening a URL and entering a shared 4-letter room code.

---

## How to Play

1. Open the game URL in your browser
2. Enter a room code and your name in the lobby
3. Share the same room code with up to 3 friends
4. When 3+ players join, the game starts automatically
5. A trivia question appears at the top of the screen
6. Jump up to the platform with your answer before time runs out
7. Fight opponents off the correct platform to steal their point
8. Highest score after 15 rounds wins

**Controls:**
- Move: ← / → Arrow keys
- Jump: Space
- Punch: X

## Architecture

```
Browser (up to 4 players)
  Godot 4 HTML5 Export
  └── GDScript game logic
  └── JavaScriptBridge → Socket.io JS client
            │
            │  WebSocket (Socket.io protocol)
            │
  Python Server (FastAPI + python-socketio)
  ├── Lobby & room management
  ├── 15-second authoritative round timer
  ├── Player position relay (~20×/sec)
  ├── Attack validation & knockback dispatch
  ├── Server-side scoring (position-based at timer end)
  └── Question bank (76 AI-generated questions)
```

**Key design decision:** The server is fully authoritative. Clients send their inputs up; the server relays state back down. Scoring is validated server-side by checking player positions at the exact moment the timer hits zero — clients cannot cheat by self-reporting a correct answer.

**Godot ↔ Socket.io bridge:** Since Godot exports to HTML5, GDScript can call JavaScript directly via `JavaScriptBridge`. The Socket.io JS client runs in the HTML page; GDScript polls a JS event queue every frame to receive server events and fires signals that the rest of the game listens to.

## Features

- **Real-time multiplayer** — up to 4 players per room, WebSocket-based
- **Physics brawler** — punch and knock opponents off platforms
- **King-of-the-hill scoring** — server checks positions at timer end, not on touch
- **No repeated questions** — question bank is shuffled at game start, questions popped in order
- **Player colors** — each player slot gets a unique color (red, blue, green, yellow)
- **Restart** — any player can restart the game after it ends, same room same players
- **HUD** — live timer (turns red at 5 seconds), live scores, round result flash, controls reminder
- **AI-generated questions** — 76 questions across 8 categories generated by Claude Haiku

---

## Question Categories

Science, History, Geography, Pop Culture, Sports, Food, Movies, Animals

Questions were generated in bulk at hackathon start using the Anthropic Claude Haiku API and saved to `questions.json`. No API calls happen during gameplay — all questions are served from the local file, keeping latency zero and cost negligible.

## How to Run Locally

**Backend:**
```bash
cd server
pip install -r requirements.txt
uvicorn index:socket_app --host 0.0.0.0 --port 3000 --reload
```

**Game:**
Open `http://localhost:3000` in up to 4 browser tabs. Each tab is one player. Enter the same room code in each tab to play together.

**Godot (editor testing):**
Open `godot/project.godot` in Godot 4. Press F5 to run locally — networking is disabled in editor mode, but physics and gameplay can be tested.

## Challenges & Solutions

**Godot ↔ Socket.io:** Godot's built-in WebSocket speaks raw WS but Socket.io uses its own protocol. Since we export to HTML5, we solved this by loading the Socket.io JS client in the HTML page and calling it from GDScript via `JavaScriptBridge.eval()`. This was the highest-risk technical piece and was validated first.

**Authoritative scoring:** Early versions let clients self-report when they were on the correct platform. This was replaced with server-side position checking at timer end — the server compares each player's last-known coordinates against the correct platform's fixed coordinates.

**Question repetition:** Using `random.choice()` could repeat questions within a game. We solved this by shuffling the full 76-question bank at game start and popping questions in order, guaranteeing no repeats within a single game.
